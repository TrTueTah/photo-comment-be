## 1. JWT Authentication in NestJS

**Decision**: Use `@nestjs/jwt` + `@nestjs/passport` with Passport JWT strategy. Issue
short-lived access tokens only (no refresh tokens for v1 simplicity).

**Rationale**: Official NestJS auth recipe. Stateless tokens work well with the REST
design. Omitting refresh tokens reduces scope while satisfying the demo requirement.

**Key decisions**:
- Token payload: `{ sub: userId, email }` — minimal, non-sensitive.
- Token lifetime: 24h (configurable via `JWT_EXPIRES_IN` env var).
- Token transmission: `Authorization: Bearer <token>` header.
- Logout: stateless — client discards the token. Server-side blocklist out of scope.

**Alternatives considered**:
- Session-based auth (express-session + cookies): stateful, requires session store —
  unnecessary complexity for a simple API.
- Refresh token rotation: better security but doubles complexity for a demo project.

## 2. Google OAuth 2.0 in NestJS

**Decision**: `passport-google-oauth20` strategy via `@nestjs/passport`. Callback URL
handled at `GET /auth/google/callback`. On success, issue a JWT (same flow as local).

**Rationale**: Standard pattern for Google OAuth in NestJS. Passport abstracts the OAuth
dance cleanly.

**Key decisions**:
- Callback URL: `http://localhost:3000/auth/google/callback` (dev); configurable via
  `GOOGLE_CALLBACK_URL` env var.
- Scopes requested: `profile`, `email`.
- Account linking: if a Google email matches an existing email/password account, the
  `googleId` is attached to that account and the user is returned a JWT — no duplicate
  accounts.
- New Google users: a User record is created with `passwordHash = null`.

**Alternatives considered**:
- Firebase Auth: vendor lock-in, overkill for a take-home demo.
- Auth0: same concern.

## 3. S3 Image Upload — Client-Direct via Presigned URLs

**Decision**: Client uploads directly to S3 using a presigned `PUT` URL. The backend
never handles the file bytes. Flow is two steps:
1. Client asks backend for a presigned URL (`POST /photos/presign`).
2. Client PUTs the file directly to S3 using that URL, then confirms to the backend
   (`POST /photos` with the S3 key).

**Packages**: `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`. No `multer`.

**Rationale**: Server never buffers file data → lower memory pressure, no upload timeout
risk on the backend, and the server scales independently of file size. This is the
standard pattern for direct-to-S3 uploads.

**Key decisions**:
- S3 bucket: configured via `AWS_S3_BUCKET` env var.
- Object key: `photos/<uuid>.<ext>` — generated by backend at presign time.
- Presigned URL type: `PutObjectCommand` signed with `getSignedUrl` from
  `@aws-sdk/s3-request-presigner`. Expiry: 300 seconds (5 minutes).
- `ContentType` is embedded in the presigned URL conditions. The client MUST send the
  matching `Content-Type` header when calling the presigned URL; S3 will reject mismatches.
- ACL / public access: bucket policy allows `s3:GetObject` for `*` (public read). The
  stored URL is `https://<bucket>.s3.<region>.amazonaws.com/<key>`.
- AWS credentials: via `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION` env
  vars (standard SDK credential chain).
- MIME type validation: backend validates `contentType` in the presign request body
  before generating the URL. Allowed: `image/jpeg`, `image/png`, `image/gif`,
  `image/webp`. Invalid types are rejected with HTTP 422 before any S3 interaction.
- File size: cannot be hard-enforced server-side with a plain `PutObject` presigned URL.
  Client-side validation (10 MB) is the primary enforcement. S3 bucket policy can add a
  secondary limit if desired, but is not required for v1.

**Alternatives considered**:
- Server-side streaming (multer → S3 PutObject): server handles all bytes; more
  backend load; rejected per user requirement for client-direct upload.
- Presigned POST (multipart form policy): supports server-enforced size via policy
  conditions but requires more complex client code. Deferred to a future version.
- `multer-s3`: still server-side; rejected for same reason.

## 4. Prisma Setup in NestJS

**Decision**: `PrismaService` extends `PrismaClient` and implements `OnModuleInit` to
call `$connect()`. Registered as a global module (`@Global()`) so all feature modules
can inject it without re-importing `PrismaModule`.

**Rationale**: Standard NestJS + Prisma pattern from official Prisma docs. Global module
avoids repetitive imports in every feature module.

**Key decisions**:
- Database URL via `DATABASE_URL` env var.
- Migrations run with `prisma migrate deploy` in production, `prisma migrate dev` in dev.
- No soft-delete or audit fields in v1 (simplicity principle).

---

## 5. File Validation (MIME type)

**Decision**: MIME type is validated in the `POST /photos/presign` DTO using
`class-validator`. The `contentType` field MUST be one of: `image/jpeg`, `image/png`,
`image/gif`, `image/webp`. Invalid values return HTTP 422 before a presigned URL is
generated.

File size is enforced client-side (10 MB). No `multer` or server-side file buffering.
The `Content-Type` header sent by the client during the S3 PUT is validated by S3 itself
(it must match the value embedded in the presigned URL conditions).

---

## 6. Input Validation Strategy

**Decision**: Global `ValidationPipe` with `whitelist: true`, `forbidNonWhitelisted: true`,
`transform: true` registered in `main.ts`.

All request bodies use DTO classes annotated with `class-validator`. Path params
validated with built-in `ParseUUIDPipe`.

## 7. Environment Variables Required

| Variable | Purpose | Example |
|----------|---------|---------|
| `DATABASE_URL` | PostgreSQL connection string | `postgresql://user:pass@localhost:5432/photodb` |
| `JWT_SECRET` | JWT signing secret | long random string |
| `JWT_EXPIRES_IN` | Token lifetime | `24h` |
| `GOOGLE_CLIENT_ID` | Google OAuth app ID | from Google Cloud Console |
| `GOOGLE_CLIENT_SECRET` | Google OAuth secret | from Google Cloud Console |
| `GOOGLE_CALLBACK_URL` | OAuth redirect URI | `http://localhost:3000/auth/google/callback` |
| `AWS_ACCESS_KEY_ID` | AWS credential | IAM user key |
| `AWS_SECRET_ACCESS_KEY` | AWS credential | IAM user secret |
| `AWS_REGION` | S3 bucket region | `ap-southeast-1` |
| `AWS_S3_BUCKET` | S3 bucket name | `photo-comment-dev` |
| `PORT` | Server port (optional) | `3000` |
